## Why bit-exact parity holds

The math is identical; only the *which CPU instruction* changes.

- Scalar inner loop accumulates `d_g: i32 += (1<<k) * (m - c)` in
  `(w outer, k inner)` order across 16 popcounts.
- v4 inner loop computes the same 16 `(m - c)` values in SIMD lanes,
  multiplies by the per-lane weight vector
  `[1,1,1,1, 2,2,2,2, 4,4,4,4, 8,8,8,8]`, and reduces with
  `_mm512_reduce_add_epi32`.
- Integer addition (mod 2³²) is associative and commutative ⇒ the
  reduction tree produces the same `i32` value as the scalar
  left-to-right sum.
- Both paths cast `d_g as f32` exactly *once*. Same input bits ⇒ same
  output bits.
- The doc-side `s_g = popcount(pos) - popcount(neg)` is an i32 computed
  identically in both paths (v4 uses scalar `popcntq` × 8, exactly like
  v3's hoisted block).
- The remaining f32 ops (`d_scale * (q_offset * s_g + q_scale * d_g)`,
  `cos * qc + sin * resi`, max-update, final NEG_INFINITY-skipped sum)
  use the same operands in the same order.

This is asserted by the `x86v4_matches_x86v3_on_production_shape` test
using `f32::to_bits()` equality (not `abs_diff_eq`).

## Expected speedup (analytical)

Production inner loop per `(j, i)` pair at `dim=128, n_words=4,
query_bits=4, n_groups=1`:

- 32 popcounts (16 `m`-popc + 16 `c`-popc)
- ~32 ANDs, 16 subs, 16 mults, 16 adds (reduction)
- ~80+ scalar µops on v3, ~20-cycle inner per Skylake-class 4-issue.

On the v4 path:

- 1 × 512-bit load of `q_planes[i, ...]` (16 i32, port 2/3)
- 2 × `_mm512_and_si512` (port 0/5)
- 2 × `_mm512_popcnt_epi32` (port 5, throughput 1/cycle each on
  Sapphire Rapids; back-to-back ≈ 2 cycles)
- 1 × sub + 1 × mullo + 1 × `_mm512_reduce_add_epi32`
  (≈ 8-10 µops total, ~10 cycles for the reduction tree)

Net: ~12-14 cycle inner per `(j, i)` vs. ~20 cycle scalar →
**~1.5-2× wall-clock speedup at the inner-loop level.** Above the
inner loop, the per-`j` doc-side hoist and the rayon doc-parallelism
are unchanged, so we don't expect amplification.

The headline 16×-popcount-throughput is *not* fully realisable here
because the inner loop is bound by `_mm512_reduce_add_epi32`'s
log-tree latency, not by popc throughput. To capture the full 16×
we'd need to keep `d_g` as a SIMD vector across multiple `j`s and
reduce only at the end — but that conflicts with the per-`j` masking
and the per-`(i, j)` `est = cos * qc + sin * resi` epilogue. Punt
to a future revision.

The wins compound on CPU benchmarks where the inner loop dominates:
LEMUR-routed CPU search at K=2000 candidates × 32 q-tokens × 32
d-tokens × 4 query_bits × 4 words ≈ 16 M popcounts per query, where
the 1.5-2× factor translates directly to QPS.

## Backward compatibility

- The build is identical for non-x86_64 targets (`#[cfg(target_arch =
  "x86_64")]` gates everything new).
- On x86_64 hosts without `avx512vpopcntdq`: `select_backend()` sees
  the missing feature and continues to return `BACKEND_X86V3` as
  before. No new code path is reached. No object-file footprint
  cost beyond a few KB of dead code (LLVM may DCE it under LTO
  since the const-prop'd `select_backend()` will never reach v4 on
  AVX2-only builds, but we don't rely on this).
- On x86_64 hosts WITH `avx512vpopcntdq` but at a non-default
  rroq158 shape (e.g. `query_bits=2`, `dim=64`): `score_pair_x86v4`
  internally tail-calls `score_pair_x86v3`, so we never regress
  unusual configs.

## How to validate at maturin time

After `cd src/kernels/shard_engine && cargo test --release`:

1. The new test `x86v4_matches_x86v3_on_production_shape` will run
   automatically on hosts with `avx512vpopcntdq` (current H100 box,
   all Sapphire Rapids/Ice Lake-server/Zen 4+ hosts).
2. On hosts without it (e.g. the published-benchmark RTX A5000 box),
   the test prints `skipping x86v4 parity test: AVX-512 VPOPCNTDQ
   unavailable` and exits 0.
3. Existing `score_pair_zeros_yields_zero`,
   `score_pair_centroid_only`, `score_pair_doc_mask_excludes_token`
   tests remain unchanged and continue to verify the scalar path.

Pre-existing parity tests at the Python level
(`tests/test_rroq158_parity.py`) also continue to validate against
the Triton reference path; once `maturin develop --release` is run
they'll exercise the v4 backend transparently on this host.
